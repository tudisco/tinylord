Why another one?

Most small apps don't need a kingdom.

A personal tool, a small-business app, a little website — they rarely need enterprise anything. Yet every time, you end up writing another server, another API, another pile of REST endpoints just to keep some data somewhere. It's a chore, and it's the same chore every time.

There are answers. None of them fit a small kingdom:

Firebase

Brilliant, borrowed

It just works — but you're renting a corner of someone else's cloud, forever. I didn't want to depend on anybody.

Appwrite

Self-hosted, but heavy

You can run it yourself, but it arrives with an entire royal court in tow. Enormous bloat for a small job.

PocketBase

The one I loved

A single binary that just works. I've reached for PocketBase a few times and adored it — but even this was more than my little apps needed.

TinyLord

Could it be smaller?

So I asked the obvious question — and kept cutting until only the essentials were left. This is the answer.

Powers of the realm

Small, but it rules well.

Everything a browser-only or small-server app usually needs from a backend — and nothing it doesn't.

🪶

Featherweight

One static binary that idles in single-digit megabytes. No runtime to install, no daemon zoo, no orchestration. Drops onto a $5 VPS or a Raspberry Pi.

🗄️

One file per database

Plain SQLite under the hood (SQLCipher, actually). Back it up by copying a file. Move it by copying a file. Nothing arcane.

📡

Realtime, built in

Subscribe to a collection and get live inserts, updates, and deletes over Server-Sent Events. The reason browser-only apps usually need a custom backend — solved.

🧩

Schemaless documents

Store JSON, query it with a small Mongo-ish filter language. No migrations, no schema wrangling, and never any SQL to write from the client.

🔒

Encrypted by default

AES-256 at rest via SQLCipher — data files, snapshots, and the control database. Your key stays server-side; it never touches config, logs, or the API.

🎫

Tokens & grants

Bearer tokens with read / write / admin grants per database. Ship a scoped token to the browser; keep the keys to the kingdom.

💾

Painless backups

One endpoint makes a clean, encrypted snapshot. Pair it with a Litestream sidecar when you want continuous point-in-time recovery.

🚫

Depends on nobody

Self-hosted, no cloud account, no vendor, no lock-in. Your data lives on your machine, encrypted, under your key. That's the whole idea.

The decree

Running in about a minute.

One binary to start, one token to hand out, and you're storing and streaming data. Here's the whole ceremony.

royal-charter.sh 
# 1 · Start it. One binary, encrypted at rest by default.
tinylord serve

# 2 · Create a database + a browser-scoped token (admin CLI or API).
tinylord db create app
tinylord admin create-user --name web
tinylord admin grant --user <ID> --db app --role write

# 3 · Write a document — no schema, no SQL, no migration.
curl -X POST localhost:8090/v1/db/app/collections/notes/documents \
  -H "Authorization: Bearer $TOKEN" \
  -d '{"title":"Hello","done":false}'

# 4 · Subscribe to live changes over SSE. This is the magic bit.
curl -N localhost:8090/v1/db/app/collections/notes/subscribe \
  -H "Authorization: Bearer $TOKEN"

Where the kingdom ends

A small lord, honestly.

TinyLord is deliberately tiny. Knowing what it won't do is how you know it's right for you.

👑
One throne, one processNo clustering, replication, or multi-node. One process per database file — simple by design.
📨
Best-effort realtimeLive updates are fast but not guaranteed delivery. Perfect for UIs; not for a financial ledger.
🔍
A small query languageEquality, comparisons, $in, and one $or. No joins, aggregation, $regex, or full-text.
🖥️
No admin UICLI and API only — that's the point. Bring your own dashboard if you want one.
🔐
No built-in TLSPut it behind a reverse proxy like Caddy or nginx for HTTPS. It expects to sit behind one.
🏰
Small kingdoms onlyServing millions across regions? You've outgrown TinyLord. Go hire an empire — with our blessing.
